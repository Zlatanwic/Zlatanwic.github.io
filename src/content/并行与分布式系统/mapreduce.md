# MapReduce：用两个函数撬动大规模数据处理

在大数据系统的发展史上，MapReduce 是一个绕不开的名字。

它最早由 Google 提出，用来解决海量网页、日志、索引、图数据和机器学习数据的离线处理问题。今天我们回头看，MapReduce 的思想并不复杂：用户只需要写两个函数，`Map` 和 `Reduce`，系统负责把任务分发到成百上千台机器上运行。

真正厉害的地方在于：它把大规模分布式计算里最麻烦的部分，比如数据切分、任务调度、机器故障、网络传输、负载均衡等，都封装到了运行时系统里。

也就是说，MapReduce 的目标不是让程序员学会写复杂的分布式程序，而是让程序员**不用写分布式程序，也能使用大规模集群**。

---

## 1. 问题背景：业务逻辑简单，分布式执行很难

Google 内部有很多数据处理任务，比如：

* 统计网页中的单词频率；
* 构建搜索引擎的倒排索引；
* 分析 Web 请求日志；
* 统计 URL 访问次数；
* 提取网页链接关系；
* 处理大规模机器学习训练数据。

这些任务从业务逻辑上看并不难。比如词频统计，无非就是遍历文档，把每个单词出现次数加起来。

如果数据只有几 MB，单机程序很容易写。但 Google 面对的是 TB 级甚至更大规模的数据，必须在几百到几千台机器上并行处理。

这时困难就来了：

* 输入数据怎么切分？
* 每台机器处理哪一部分？
* 中间结果放在哪里？
* 相同 key 的数据怎么汇总？
* 某台机器挂了怎么办？
* 某些机器特别慢怎么办？
* 网络带宽不够怎么办？
* 程序员是不是每次都要手写 RPC、重试和调度逻辑？

MapReduce 的出发点就是：这些底层问题不应该让每个业务程序重复解决。系统应该提供一个统一抽象，让用户只关注业务逻辑。

---

## 2. 核心思想：Map 负责“展开”，Reduce 负责“聚合”

MapReduce 的编程模型非常简单。

一个 MapReduce 任务的输入是一组 key/value pair，输出也是一组 key/value pair。用户需要提供两个函数：

```text
Map(k1, v1) -> list(k2, v2)

Reduce(k2, list(v2)) -> list(v2)
```

可以这样理解：

* `Map`：处理一条输入记录，产生若干中间 key/value；
* `Reduce`：处理同一个 key 下的所有 value，得到最终结果。

最经典的例子是词频统计。

假设有两篇文档：

```text
doc1: hello world
doc2: hello mapreduce
```

Map 函数会输出：

```text
<hello, 1>
<world, 1>
<hello, 1>
<mapreduce, 1>
```

系统会自动把相同 key 的数据聚到一起：

```text
hello -> [1, 1]
world -> [1]
mapreduce -> [1]
```

Reduce 函数再对每个 key 的 value 求和：

```text
<hello, 2>
<world, 1>
<mapreduce, 1>
```

用户写的只是 Map 和 Reduce，至于中间数据怎么传输、怎么排序、怎么分配到不同机器，全部由 MapReduce 系统负责。

这就是它最重要的抽象能力。

## 3. MapReduce 的完整执行流程

一个 MapReduce 作业大致可以分为以下几个阶段。

### 3.1 切分输入数据

系统首先把输入文件切成很多小块，论文中通常是 16MB 到 64MB 一个 split。每个 split 对应一个 map task。

如果输入数据有 1TB，每个 split 是 64MB，那么大约会产生 15000 个 map task。

这样做有两个好处。

第一，任务可以并行执行。
第二，任务粒度足够细，方便负载均衡和失败恢复。

### 3.2 Master 分配任务

MapReduce 采用 master-worker 架构。

其中一个进程是 master，负责任务调度和元数据管理；其他进程是 worker，负责真正执行 map task 或 reduce task。

master 会维护每个任务的状态：

```text
idle
in-progress
completed
```

当某个 worker 空闲时，master 就给它分配一个 map task 或 reduce task。

### 3.3 Map Worker 读取数据并产生中间结果

被分配到 map task 的 worker 会读取对应的 input split，然后调用用户写的 Map 函数。

Map 函数产生的中间 key/value pair 先缓存在内存里，随后写到本地磁盘。

这里有一个关键细节：map 输出不是直接写到分布式文件系统，而是写到 map worker 的本地磁盘。

原因很简单：map 输出只是临时中间结果，不一定需要长期保存。写本地磁盘比写分布式文件系统更快，也能减少网络开销。

### 3.4 中间结果按 key 分区

MapReduce 会根据中间 key，把 map 输出划分成 R 个分区。常见方式是：

```text
hash(key) mod R
```

其中 R 是 reduce task 的数量。

如果 R = 100，那么每个 map task 会把自己的中间结果分成 100 份，每一份对应一个 reduce task。

这样可以保证：同一个 key 的数据最终会被送到同一个 reduce task。

### 3.5 Shuffle：Reduce Worker 拉取中间数据

当 map task 完成后，master 会记录这些中间文件的位置。

reduce worker 会从 master 那里获得中间文件位置信息，然后通过网络从各个 map worker 的本地磁盘读取属于自己的那部分数据。

这个阶段通常叫 shuffle。

Shuffle 是 MapReduce 中代价最高的阶段之一，因为它涉及大量网络传输。

### 3.6 Sort：按 key 排序

reduce worker 拉取完数据后，会对中间 key 进行排序。

排序的目的不是为了好看，而是为了让相同 key 的数据连续出现。这样 reduce worker 就可以依次遍历每个 key，并把对应 value 列表交给用户写的 Reduce 函数。

如果中间数据太大，内存放不下，系统会使用外部排序。

### 3.7 Reduce 产生最终输出

Reduce 函数处理每个 key 以及它对应的所有 value，输出最终结果。

每个 reduce task 会生成一个输出文件，所以整个 MapReduce 作业最终会产生 R 个输出文件。

论文中也特别提到，用户通常不需要把这些文件再合并成一个大文件。因为后续的 MapReduce 作业或者分布式应用一般可以直接处理分片文件。

## 4. 为什么 MapReduce 能容错？

在几千台普通机器组成的集群中，机器故障不是小概率事件，而是日常现象。

MapReduce 的容错思路非常直接：**失败了就重跑**。

### 4.1 Worker 挂了怎么办？

master 会周期性 ping 每个 worker。如果某个 worker 长时间没有响应，就认为它失败。

如果失败 worker 上有正在运行的任务，master 会把这些任务重新标记为 idle，然后分配给其他 worker 执行。

如果一个 worker 已经完成了 map task，但后来挂了，这些 map task 也需要重新执行。因为 map 输出存储在该 worker 的本地磁盘上，机器挂了以后这些中间结果就读不到了。

但是，如果 worker 已经完成了 reduce task，一般不需要重跑。因为 reduce 输出写到了全局分布式文件系统中。

所以 MapReduce 里有一个很重要的区别：

```text
Map 输出：临时数据，写本地磁盘
Reduce 输出：最终数据，写分布式文件系统
```

### 4.2 为什么重跑是安全的？

如果用户写的 Map 和 Reduce 是确定性的，那么同一个输入重跑多次，结果应该一样。

MapReduce 利用这个性质实现了简单有效的容错。

每个任务运行时先写临时文件，只有任务完成后才通过原子操作提交结果。这样即使同一个任务被执行多次，最终也只会有一个结果生效。

这也是 MapReduce 鼓励用户写确定性函数的重要原因。

---

## 5. 为什么要关心数据本地性？

在大规模集群中，网络带宽往往比本地磁盘带宽更稀缺。

如果一个 map task 要处理的数据在机器 A 上，却被调度到机器 B 上执行，那么数据就要通过网络从 A 传到 B。数据量一大，网络很快会成为瓶颈。

MapReduce 的优化方式是：尽量把 map task 调度到存有对应输入数据副本的机器上。

这就是常说的：

```text
把计算移动到数据旁边，而不是把数据移动到计算旁边。
```

Google 的 GFS 会把文件切成 block，并保存多个副本。MapReduce master 知道每个 block 的位置，于是可以优先把 map task 分配给拥有该数据副本的 worker。

如果无法做到完全本地，也尽量调度到同一个机架或同一个交换机附近的机器上。

这个优化看似简单，但对大规模数据处理非常关键。

---

## 6. Straggler：最慢的几个任务决定整体时间

大规模分布式系统中还有一个常见问题：straggler。

所谓 straggler，就是执行特别慢的任务。它可能来自：

* 某台机器磁盘变慢；
* 某台机器同时运行了其他任务；
* 网络出现抖动；
* 硬件异常；
* 系统配置问题。

即使 99.9% 的任务都完成了，只要最后几个任务迟迟不结束，整个作业就不能结束。

MapReduce 的解决方案是 backup task，也可以叫推测执行。

当一个作业接近完成时，master 会为剩余还没完成的任务启动备份执行。原任务和备份任务谁先完成，就采用谁的结果。

这会多消耗一点资源，但可以显著减少尾部延迟。

论文中的实验表明，对于排序任务，如果关闭 backup task，整体完成时间会增加 44%。这个结果说明，在大规模集群里，处理 straggler 是非常必要的。

---

## 7. Combiner：在 Map 端先做一次局部聚合

MapReduce 中还有一个很实用的优化：Combiner。

继续看词频统计。

如果某个 map task 处理的文本中出现了很多次 `the`，Map 会产生大量：

```text
<the, 1>
<the, 1>
<the, 1>
...
```

如果这些数据全部通过网络发送给 reduce worker，会浪费大量带宽。

Combiner 的作用就是在 map worker 本地先做一次局部合并：

```text
<the, 1>
<the, 1>
<the, 1>
```

先变成：

```text
<the, 3>
```

然后再发送给 reduce worker。

这样可以大幅减少 shuffle 数据量。

不过 Combiner 不是任何任务都能用。它要求操作通常满足结合律和交换律，比如：

* sum；
* count；
* max；
* min。

如果是 average，就不能简单地把局部平均值再平均，否则结果可能是错的。正确做法通常是传递 `(sum, count)`，最后再计算平均值。

---

## 8. MapReduce 适合哪些任务？

MapReduce 适合的问题通常有一个共同特点：

> 可以先对大量记录独立处理，再按 key 聚合。

典型任务包括：

### 8.1 词频统计

Map 输出 `<word, 1>`，Reduce 求和。

### 8.2 日志统计

Map 输出 `<URL, 1>`，Reduce 统计每个 URL 的访问次数。

### 8.3 倒排索引

Map 输出 `<word, document_id>`，Reduce 合并每个 word 对应的文档列表。

这是搜索引擎索引构建的核心形式。

### 8.4 反向链接图

Map 处理每个网页，输出 `<target_url, source_url>`，Reduce 收集所有指向同一个 target 的 source。

### 8.5 分布式排序

Map 输出 `<sort_key, record>`，系统按 key 分区和排序，Reduce 原样输出。

## 9. MapReduce 的性能表现

论文中评估了两个任务：Grep 和 Sort。

Grep 任务扫描约 1TB 数据，搜索一个罕见模式。输入被切成约 15000 个 map task，最终输出到一个文件。实验中，扫描速率峰值超过 30GB/s，整个任务大约 150 秒完成。

Sort 任务排序约 1TB 数据。它使用 15000 个 map task 和 4000 个 reduce task。Map 提取排序 key，Reduce 原样输出。最终任务用 891 秒完成，接近当时 TeraSort benchmark 的优秀结果。

论文还专门测试了两种情况：

第一，关闭 backup task。
结果排序任务从 891 秒变成 1283 秒，耗时增加 44%。

第二，人为杀掉 200 个 worker。
最终任务仍然完成，总耗时只比正常情况增加 5%。

这些实验说明，MapReduce 不仅能扩展到大规模集群，而且面对慢机器和机器故障时仍然比较稳健。

## 10. MapReduce 在 Google 的实际应用

MapReduce 在 Google 内部被广泛用于：

* 大规模机器学习；
* Google News 聚类；
* Froogle 数据处理；
* 热门搜索查询统计；
* 网页属性抽取；
* 地理位置信息提取；
* 大规模图计算；
* 搜索引擎索引构建。

论文中特别提到，Google 用 MapReduce 重写了生产级 indexing system。原始网页数据超过 20TB，索引过程由 5 到 10 个 MapReduce 作业组成。

使用 MapReduce 后，系统带来了几个好处：

第一，代码大幅简化。某个阶段从大约 3800 行 C++ 代码减少到大约 700 行。

第二，系统更容易修改。旧系统中需要几个月的改动，在新系统中几天就可以完成。

第三，运维更简单。机器故障、慢机器、网络抖动等问题大多由 MapReduce 自动处理。

这说明 MapReduce 的价值不仅在性能，也在软件工程效率。

## 11. MapReduce 的设计哲学

MapReduce 最值得学习的地方，不是 Map 和 Reduce 两个函数本身，而是它背后的系统设计哲学。

### 11.1 用受限模型换取自动化

MapReduce 不试图表达所有计算。

它只适合一类可以拆成 Map 和 Reduce 的任务。这个限制看似降低了通用性，但正是因为模型受限，系统才能自动完成并行化、调度和容错。

这是一种经典系统设计思路：

```text
限制表达能力，换取可扩展性和可靠性。
```

### 11.2 默认机器会失败

MapReduce 不假设机器永远可靠。相反，它默认 worker 会挂、任务会失败、机器会变慢。

因此它从一开始就把失败恢复设计进系统里。

这种思路对今天的大规模系统仍然非常重要。

### 11.3 网络是稀缺资源

MapReduce 的很多优化都围绕减少网络传输展开：

* map task 尽量调度到数据所在机器；
* map 输出先写本地磁盘；
* 使用 combiner 减少 shuffle 数据量；
* 中间数据只在需要时被 reduce worker 拉取。

这说明在分布式系统中，计算不是唯一瓶颈，网络和 I/O 往往更关键。

### 11.4 尾部延迟必须处理

大规模任务的完成时间往往不是由平均速度决定，而是由最慢的几个任务决定。

Backup task 的设计说明，系统有时可以用少量冗余计算换取更稳定的整体完成时间。

---

## 12. MapReduce 的局限性

虽然 MapReduce 非常成功，但它也不是万能的。

### 12.1 不适合低延迟任务

MapReduce 更适合离线批处理，不适合实时查询或低延迟服务。它的启动、调度、shuffle 和写文件都有较大开销。

### 12.2 不适合复杂迭代计算

很多机器学习和图算法需要反复迭代。原始 MapReduce 每次迭代都要读写分布式文件系统，开销较大。

这也是后来 Spark 等系统引入内存计算的重要原因。

### 12.3 编程模型表达能力有限

MapReduce 适合“独立处理 + 按 key 聚合”的任务，但对于复杂的数据流、多阶段依赖、交互式查询，它的表达能力不够自然。

### 12.4 Shuffle 成本高

一旦中间数据很大，shuffle 会带来大量网络传输和磁盘 I/O，成为性能瓶颈。

---

## 13. 总结

MapReduce 是一个非常经典的大规模数据处理框架。

它的核心思想可以概括为：

```text
用户写 Map 和 Reduce；
系统负责并行化、调度、shuffle、排序、容错和负载均衡。
```

它的成功来自三个方面。

第一，模型足够简单，普通程序员也能使用大规模集群。

第二，很多真实世界的大数据任务都可以表达成 MapReduce 形式。

第三，它的系统实现充分考虑了工程现实，包括机器故障、慢任务、网络瓶颈和数据本地性。

MapReduce 最重要的启发是：

> 一个好的分布式系统，不一定要给用户最强的表达能力，而是要在合适的抽象边界上隐藏复杂性。

这也是为什么 MapReduce 虽然已经诞生多年，但它的思想仍然深刻影响着今天的大数据系统、分布式计算框架和云计算平台。

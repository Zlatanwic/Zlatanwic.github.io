# 个人知识库的目录应该怎么设计

知识库最容易失败的地方，不在于“写得不够多”，而在于“以后根本找不到”。所以目录设计不是装饰，而是维护能力本身。

## 我建议从三个问题出发

### 这份内容以后会被怎样使用

如果内容主要是反复查阅，就应该按主题和任务组织，而不是按日期组织。

### 这份内容会不会继续长

会继续长的内容，不要塞进太深的层级里。层级一深，维护意愿就会快速下降。

### 别人第一次进来能不能知道从哪开始

每个大类最好都有一个 index 页，负责解释这个分类放什么、不放什么，以及推荐阅读顺序。

## 一个足够轻的结构

- `notes/` 作为总入口
- `writing/` 放写作相关工作流
- `tools/` 放工具手册和维护记录
- `thinking/` 放方法论与思维模型

这个结构的优点是：

- 名字直观
- 可以继续扩展
- 不会一开始就把目录设计得过度复杂

## 判断一个目录好不好，有个很简单的标准

三个月后你回来看，还愿不愿意继续往里写。

如果不愿意，说明目录不是在帮助你，而是在增加摩擦。
